### 2026-06-11 — MVP slice 2: global Start/Stop hotkey via Carbon (not Accessibility)
- **Agent**: Claude Code (confirmed by Yuval)
- **Decision**: The global Start/Stop hotkey (default ⌃⌥⌘R) uses Carbon `RegisterEventHotKey` — one registered chord, fires system-wide, routed to the app's event target. No dependency. `GlobalHotKey` in `INMeetingsCore/Input`.
- **Rationale**: `RegisterEventHotKey` registers a single specific chord and cannot observe any other input, so it needs **no Accessibility permission**. The alternative — `NSEvent.addGlobalMonitorForEvents` — receives every keystroke and is gated behind Accessibility: the scariest onboarding prompt, an enlarged privacy surface (keystroke-observation capability) for a confidential founder/deal recorder, and re-grant fragility tied to the code signature. We lose nothing — a single global chord is exactly what `RegisterEventHotKey` is for. Permission-light is an ADR-011 requirement.
- **Alternatives considered**: NSEvent global monitor + Accessibility (rejected — permission/privacy cost for a one-key feature); SwiftUI menu `.keyboardShortcut` (rejected — only fires when the app is frontmost, not global); SPM packages soffes/HotKey or sindresorhus/KeyboardShortcuts (deferred — both wrap the same Carbon API; a package buys a user-customizable recorder UI the MVP doesn't need, at the cost of a dependency — revisit if shortcut customization lands in settings).
- **Consequences**: fixed default chord for now; user customization is a later settings feature (would likely adopt KeyboardShortcuts then).
- **Also this slice (minor, no separate ADR)**: (a) the live recording timer lives in the **menu-bar status-item label** (model-driven 1s tick), not the dropdown — a `.menu`-style `MenuBarExtra` is a native `NSMenu` that won't tick a SwiftUI `Text(_, style:.timer)`; (b) the onboarding TCC wizard moved from slice 2 to **slice 3**, since the System Audio Recording + Microphone grants are only testable once capture exists; (c) local dev signs with **Apple Development** (sufficient to avoid the P2 TCC digital-silence gotcha) — **Developer ID + notarization deferred to Phase 5** rollout.

### 2026-06-11 — P2 prototype: dual-track capture VERIFIED (+ interleaved-write gotcha)
- **Agent**: Claude Code
- **Decision**: P2 (Core Audio process tap + AVAudioEngine mic → two WAVs) is verified live on M4/macOS 26.5 with **only the System Audio Recording permission** (no Screen Recording). Confirms ADR-002's capture architecture. MVP capture module to be built on this.
- **Rationale**: End-to-end proof — `✅ real audio (peak −2.7 dB)`; transcribing both tracks showed clean separation (mic = Hebrew test sentence, system = the English video, no cross-bleed). 1891 IOProc callbacks over 20s.
- **Implementation gotcha (must respect in MVP)**: the tap delivers **interleaved float32**; `AVAudioFile(forWriting:settings:)` defaults to non-interleaved → `write(from:)` fails with OSStatus **−50** and the file is silent despite the tap working. Fix: create the file with the tap's exact processing format (`commonFormat` + `interleaved:true`). Also: System Audio Recording TCC attaches to the **responsible process** — an unsigned binary launched headlessly gets silence; the real (Developer-ID-signed) app avoids this.
- **Consequences**: capture is de-risked. Remaining Phase-0 item: P4 (in-person diarization).

### 2026-06-11 — Recording modes: manual capture + in-person meetings (ADR-011)
- **Agent**: Claude Code (requirements from Yuval)
- **Decision**: Added **manual Start Recording** (menu-bar + global hotkey) and a first-class **in-person (mic-only) capture profile**. Manual start **auto-picks the profile** from the P3 audio-process detector (live call → dual-track; else → mic-only). In-person is **manual-trigger only** (no calendar auto-prompt) but gets **per-speaker diarization from v1**. New ADR-011; ADR-001/002/003/005 + DESIGN + IMPLEMENTATION_PLAN updated; `meeting_type` added to the package; new de-risk prototype **P4 (in-person diarization)**.
- **Rationale**: Manual covers the "auto-detect missed it" fallback and is the only viable trigger for face-to-face meetings (no app/remote audio to detect). In-person puts everyone on one mic, so the call-time 2-track me/them shortcut doesn't apply — single-track multi-speaker diarization is required and is a distinct, harder quality axis.
- **Alternatives considered** (Yuval chose): calendar auto-prompt for in-person (rejected — manual only); merged in-person transcript for v1 (rejected — per-speaker from v1); manual always asks call-vs-in-person (rejected — auto-pick from audio signal).
- **Consequences**: in-person diarization quality (Hebrew DER, overlap, room acoustics) is unproven → P4 prototype + dashboard manual speaker assignment as backstop. Global hotkey needs a permission-light mechanism. Downstream skills must handle packages with no `audio_system.wav` and all speakers on the mic track.

### 2026-06-11 — P3 prototype: call detection via Core Audio process I/O (not app/tab heuristics)
- **Agent**: Claude Code (course-corrected by Yuval)
- **Decision**: Meeting detection's primary signal is **Core Audio per-process audio I/O** — a process with **both input and output audio running** is a live call, identified by `kAudioProcessPropertyBundleID`. ADR-001 revised. The frontmost-app + AppleScript-tab-URL approach is dropped to optional enrichment.
- **Rationale**: Verified live on M4/macOS 26 — the app/tab approach **failed** (missed a backgrounded Google Meet call; required Automation TCC). The audio-process approach is how real call-recorders work: app-agnostic (Zoom/Meet/WhatsApp/Teams uniformly), **no Automation/Accessibility/Screen-Recording permission**, frontmost-independent, and it rejects one-way playback. Test: YouTube → `armed=no`; Meet w/ mic → `armed=YES CALL in: Google Chrome`.
- **Alternatives considered**: device-level mic-in-use (AirPods blind spot, no app id — rejected); frontmost-app + tab-URL (fragile, needs Automation, missed backgrounded calls — rejected as primary); browser extension (unnecessary given the audio signal).
- **Consequences**: API is macOS 14.2+ (fine — floor is 26). Edge case: a call muted-from-start is output-only and looks like playback → mitigate with a "seen-bidirectional" latch + calendar context (minor follow-up). Detection needs no TCC; capture (ADR-002/P2) still needs the audio-recording grant.

### 2026-06-11 — P1 prototype: context injection is post-correction, not prompt biasing
- **Agent**: Claude Code
- **Decision**: Based on the P1 ASR benchmark on real founder-meeting audio (M4/macOS 26, whisper.cpp Metal + ivrit-ai turbo GGML), the context-injection mechanism is **deterministic entity post-correction**, not Whisper `initial_prompt` biasing. ADR-003 and ADR-004 amended accordingly. Evidence: `pipeline/benchmarks/P1-FINDINGS.md`.
- **Rationale**: Measured — `initial_prompt` with Latin proper nouns *regressed* names the model otherwise got right (אלגוליון/Algolion → אלכוהוליון/"alcohol-ion", 6×); Hebrew-script terms were neutral; neither fixed the hardest term ("IN Venture"). The forced-Hebrew fine-tune won't emit Latin and its prompt-following is weak (matches the research's catastrophic-forgetting caveat). Post-correction (canonical entity + variant spellings → whole-token replace) was validated: deterministic, no regressions, and yields the canonical Latin spellings the Claude skills want.
- **Alternatives considered**: Latin-script `initial_prompt` (regresses — rejected); Hebrew-script `initial_prompt` (neutral — kept only as an optional light primer, no Latin terms); model fine-tuning for code-switching (out of scope for now).
- **Consequences**: ADR-004 emits a correction vocabulary (canonical + variants) rather than a ranked prompt word-list; the CI check becomes a post-correction fixture test; a fuzzy/edit-distance pass for unseen variants is the next refinement. On-device default is confirmed (speed + quality both pass).
- **Also confirmed GO**: on-device Hebrew ASR is fast (RTF ≈ 0.10, ~10× realtime) and high quality on the team hardware floor; beats Timeless on speaker attribution (Timeless gave only generic Speaker 1/2/3).

### 2026-06-11 — Design-phase ADRs (ADR-001 … ADR-010)
- **Agent**: Claude Code
- **Decision**: Recorded the full design in `DESIGN.md` + `adr/`. One-line summaries (each links to its ADR):
  - **[ADR-001](adr/ADR-001-call-detection-trigger-ux.md)** — Multi-signal detection (running-app ∪ mic-in-use ∪ calendar ∪ browser-tab via AppleScript); **prompt-to-record** (not silent auto) with a **ring-buffer** so pre-confirm audio isn't lost; non-activating `NSPanel` over fullscreen; menu-bar `LSUIElement`.
  - **[ADR-002](adr/ADR-002-capture-pipeline.md)** — **Core Audio process tap** (remote/system, per-process) + **AVAudioEngine** (mic) as **two WAV tracks**; raw mic + **offline AEC**; **video off by default** (SCK/HEVC opt-in); mic-idle-debounced auto-stop. No virtual driver, no Screen Recording TCC for audio.
  - **[ADR-003](adr/ADR-003-transcription-engine.md)** — Primary **`ivrit-ai/whisper-large-v3-turbo`** on whisper.cpp Metal (GGML), MLX optional; `initial_prompt` biasing (≤224 tokens, terms last, pinned revision + regression test); **senko** diarization on the system track only; `transcript.json`+`.txt`; **Soniox** opt-in cloud fallback; Vision OCR for slides (opt-in).
  - **[ADR-004](adr/ADR-004-context-assembler.md)** — Match meeting → Calendar event → Saventa + Dealigence → **ranked biasing vocabulary** + `context.md` (priors only); graceful degradation when unmatched.
  - **[ADR-005](adr/ADR-005-context-package-contract.md)** — Freeze the package schema as the contract; add a **`--package <path>` local-input mode** to `saventa-summary`/`generate-mom`/`process-meeting` alongside the Timeless path; richer fields enable Hebrew-name, slide, and confidence improvements.
  - **[ADR-006](adr/ADR-006-storage-drive-sync.md)** — Local **SQLite + per-meeting folder** (cache); **Google Shared Drive via per-user OAuth** as durable source of truth; **company-first** `/<Company>/<YYYY-MM-DD-…>/` layout; write-through upload (no sync client).
  - **[ADR-007](adr/ADR-007-dashboard.md)** — **SwiftUI window in the same app** over SQLite (FTS5 search); list/detail/play/open-in-Drive/re-run-skill. No separate web app.
  - **[ADR-008](adr/ADR-008-claude-auto-trigger.md)** — Headless **`claude -p` chain**, configurable per meeting, **default one-click**; meeting-type routing decides which skills run; `process-meeting`'s human-review gate kept.
  - **[ADR-009](adr/ADR-009-app-architecture-ipc.md)** — **Swift menu-bar app + bundled Python pipeline**, **file-based job queue + JSON status** IPC (simple/observable/resumable); **internal Developer-ID signing** (not App Store) to allow the audio-permission SPI; onboarding wizard for the 4 TCC grants + Drive OAuth.
  - **[ADR-010](adr/ADR-010-privacy-consent.md)** — On-device default; only the team Drive (and opt-in cloud/Claude) leave the Mac; consent metadata + jurisdiction nudge (inform, don't block) + retention controls + **do-not-record defaults for internal events**. Six items flagged for **counsel**.
- **Rationale**: Per the design brief and `RESEARCH.md`; load-bearing claims verified against primary sources (Apple Core Audio docs, ivrit.ai leaderboard, model licenses).
- **Alternatives considered**: Documented per-ADR (e.g., SCK-for-audio rejected for the monthly-nag/Screen-Recording-TCC cost; cloud-ASR-primary rejected — no Hebrew quality win + confidentiality cost; service-account Drive auth rejected in favor of per-user OAuth).
- **Consequences**: Code-switching ASR quality and the no-monthly-nag tap claim are unproven until prototyped; skills in `~/repos/claude-skills` need a coordinated `--package` change; internal signing means maintaining Developer ID + notarization.

### 2026-06-11 — MVP slice 4c: diarizer = senko (bake-off-validated on Hebrew), profile-aware speaker attribution
- **Agent**: Claude Code (confirmed by Yuval)
- **Decision**: **senko** (CoreML/ANE, MIT) is the diarizer — confirming ADR-003's choice, but now settled by **data, not assumption**. Ran a senko-vs-pyannote **bake-off** on real Hebrew founder-meeting audio (`prelligence`): senko matched **Timeless ground-truth speaker counts at both scales** (2 on the 6-min clip, **3 on the full 51-min**) at RTF ~0.001–0.004. Speaker attribution is **profile-aware** (`attribute_speakers` in `__main__`): **call** → mic = `"Me"` (internal) + diarize the **system** track into external `"Them"` (1 speaker) / `"Speaker N"` (≥2); **in-person** → diarize the **mic** track into `"Speaker N"`. Each segment is attributed to the diarized speaker by **max temporal overlap** (`diarize.assign_speakers`); a `speakers[]` table (`{id, side, track}`) + `diarized` flag are added to `transcript.json`. Diarization **degrades gracefully** (coarse labels + `diarized:false`) rather than failing the transcript.
- **Rationale**: senko's only headline edge — raw ANE speed — is **moot for our batch pipeline** (whisper dominates wall-clock), while its feared weakness (embeddings trained on English+Mandarin → unvalidated Hebrew) **did not materialize**: it matched ground truth twice. pyannote installed cleanly (torch 2.12/MPS) but its `community-1` pipeline is **HuggingFace-gated** — making it primary would bake a per-user HF token + model-terms acceptance into onboarding for all ~5 users of an on-device tool. Rejected as primary on that deployment-friction; kept as the documented fallback (ADR-003).
- **Alternatives considered** (Yuval chose the bake-off, then "just use senko"): pyannote `community-1` (gated; fallback only); FluidAudio Swift-native CoreML (deferred — larger architecture change, would remove the Python diarization hop); cloud diarization Soniox/Deepgram (opt-in only — ADR-010).
- **Consequences**: senko requires **Python <3.14** (system `python3` is 3.14) → the pipeline is pinned to a **3.11 venv** (`pipeline/.venv`, deps in `pyproject.toml` + `uv.lock`), and **`JobBridge.defaultPython` now spawns `.venv/bin/python`** (still overridable via `IN_MEETINGS_PYTHON`; Phase 5 bundles a sealed env). DER (turn-boundary accuracy) is sane-by-eye but **not numerically measured** — count + attribution are validated against Timeless; a DER harness vs a hand-labeled reference is a follow-up. The **call path** was live-verified on a *synthetic* 2-track meeting (no real separated dual-track recording exists yet — re-verify on a real multi-party call when one is recorded).
- **MVP sign-off (Yuval, 2026-06-11)**: diarization is **accepted for the MVP without a live multi-party-call test** — Yuval can't record one right now and will flag it if it's not good enough once real calls run. To make that judgeable, every job's stdout+stderr is captured to a **per-meeting `pipeline.log`** (`JobBridge` redirects the child; `__main__` emits a `diarization profile=… diarized=… speakers=… talk_s=…` summary line), so real calls leave a reviewable trail. **⏳ PENDING: confirm multi-party-call diarization quality on a real recording** (tracked in `HANDOFF.md`).

### 2026-06-14 — Evaluated `island-io/mila`: continue ours, harvest commodity wins (Apache-2.0)
- **Agent**: Claude Code (decision by Yuval)
- **Decision**: Evaluated `island-io/mila` — a mature, just-open-sourced **Apache-2.0** macOS Hebrew transcription app (Island Technology / Uri Harduf). **Do NOT fork.** Continue our architecture and **harvest four commodity features** with attribution: (1) model-download-on-launch, (2) packaging + Sparkle auto-update, (3) "Record now" prompt overlay, (4) polished SwiftUI dashboard views. Full plan: `~/.claude/plans/i-have-a-questions-cached-lark.md`.
- **Rationale**: mila's mature parts are either undifferentiated commodity (UI, model download, packaging — worth lifting) or **incompatible with our hard constraints**: it captures via **ScreenCaptureKit** (needs Screen-Recording TCC + a placeholder video stream), **mixes mic+system into one mono track** (which kills our "Me vs Them" attribution), and diarizes with **pyannote + bundled torch** (its changelog is ~10 releases of torch/notarization firefighting — independent validation of our **senko** choice). Our moat (context injection, dual-track attribution, the Claude-skill package, Drive sync) is **0% built in mila**, so forking buys no shortcut while forcing us to re-fight two architectures we deliberately rejected. Apache-2.0 lets us lift specific files with attribution.
- **Alternatives considered** (Yuval chose "continue ours + harvest"): fork mila and bolt our features on (rejected — re-fight ScreenCaptureKit→Core Audio, mono→dual-track, pyannote→senko on a foreign ~80-file codebase); ignore mila entirely (rejected — reinvent the model-download/packaging/auto-update mila already nailed).
- **Consequences**: new Apache-2.0 attribution surface (`NOTICE`, `THIRD_PARTY_NOTICES.md`, `licenses/mila-LICENSE-2.0.txt`, per-file `// Adapted from Mila …` headers). Harvests are **additive** and must not block the moat work (slice 5 schema, slice 6 Drive, Phase 2 context assembler). **Harvest 1 (model download on launch) implemented this session.** mila's CoreML/ANE encoder is **inert for us** (the Homebrew `whisper-cli` is built without CoreML) — deferred to a Phase-5 bundled `WHISPER_COREML` build. We keep the `ivrit-large-v3-turbo` model (not mila's full `large-v3`) to preserve the P1 WER/RTF baseline. Harvest 2 (Sparkle) is **gated on Developer ID + notarization + EdDSA keys** (Phase-5 prerequisites).

### 2026-06-14 — Call video capture (ScreenCaptureKit, on by default) + "meeting done?" auto-stop prompt
- **Agent**: Claude Code (decisions by Yuval, during H3 live testing)
- **Decision**:
  1. **Record call video, ON by default for the `call` profile.** Add a ScreenCaptureKit **video-only** stream (`capturesAudio = false` — audio stays on our Core Audio dual-track path) scoped to the **detected call app's window** (SCK per-app `SCContentFilter`), written as **HEVC** alongside `mic.wav` / `system.wav`. Reuses the per-app SCK pattern (cf. mila's `SystemAudioRecorder`, but a real video stream, not a 2×2 placeholder). The **in-person** profile stays audio-only.
  2. **Auto-stop is a PROMPT, not a silent stop.** On the detector's **armed→idle** edge while a call recording is in progress, float the H3 Liquid Glass card as "Meeting ended — stop & process?". **Debounced** (call must stay idle ~10–15 s to ride out network blips) and **safe-by-default**: if ignored, keep recording — never silently drop a meeting.
  3. **Sequencing**: both land **after the core spine** (slice 5 package schema → slice 6 Drive → Phase 2 context injection). Additive; must not block the moat.
- **Rationale**: video gives the full meeting record (participants + shared screen/slides) the VC workflow wants; **window-only** capture keeps unrelated/sensitive screen content out. Prompt-to-stop avoids silently ending a recording mid-meeting on a transient audio drop.
- **Consequences (important)**: video-on-by-default **reverses the call path's "permission-light / no Screen Recording TCC" posture** (ADR-002 / P3) — **every user must now grant Screen Recording** (the ADR-009 onboarding/TCC wizard gains this grant; detection + audio still need no Screen Recording). **Privacy** (ADR-010): all calls are filmed by default → revisit consent/retention defaults + the do-not-record list; **flag for counsel**. **Storage** grows to GB-scale/meeting → needs a retention/size cap (cf. mila's storage-cap setting). **Amends ADR-002** (capture adds an SCK video track), **ADR-009** (onboarding +Screen Recording), **ADR-010** (video consent/retention) — update those ADRs when the slice is built.
- **Alternatives considered** (Yuval chose): opt-in video (rejected — wants it always for calls); full-display capture (rejected on privacy — chose call-window-only); silent auto-stop (rejected for MVP — chose prompt-for-approval).

### 2026-06-14 — MVP slice 5: context-package contract frozen (`schema/`) + SQLite index (GRDB)
- **Agent**: Claude Code (directive from Yuval: "work by the plan order")
- **Decision**: Froze the ADR-005 context-package contract as machine-readable **JSON Schema** in `schema/` (`transcript.schema.json` + `metadata.schema.json` + `README.md`) and added the ADR-006 local **SQLite index via GRDB**. Sub-decisions:
  - **`transcript.json` shape = ADR-005 (option B1)**: pipeline `segments[]` → `utterances[]`; `start_ms`/`end_ms` int-ms → `start`/`end` float-**seconds**; inline `speaker` label → `speaker_id` referencing `speakers[].id`; added `confidence` (null at MVP) + provenance `engine`/`model_revision`/`biased`. Kept additive extras (`speakers[].track`, top-level `profile`/`diarized`/`meeting_id`) — consumers ignore unknowns.
  - **The Python pipeline is the single writer of the package** (ADR-009). Swift hands record-time facts via an extended `job.json` (`started_at`/`ended_at`/`capture_source_app`); sample-rate + durations are read from the WAVs (soundfile), and `video` defaults off (V1). `metadata.py` merges these with transcription facts; Phase 2's context assembler (also Python) extends `metadata.json` in place.
  - **Forward-compatible by design**: `metadata.json.schema_version` = "1.0"; every Phase-2 / slice-6 / V1 field is present-but-null/empty at MVP (calendar/CRM/consent, `video`, `confidence`) so freezing now doesn't force a breaking bump when those land.
  - **Meeting id = the local timestamp folder** (`yyyy-MM-dd_HH-mm-ss`, company-independent); company is metadata and only drives the Drive folder path (slice 6). The immutable id never re-keys on a late company match.
  - **SQLite index = GRDB** (ADR-006) — the **first external dependency** in the previously dependency-free `INMeetingsCore` SPM package, pinned to the 6.x line (Swift 5.9 toolchain). `MeetingStore.indexPackage(at:)` upserts a `meeting` row (idempotent) when the pipeline reports `done`; FTS5 + the dashboard are deferred to H4.
  - **Golden fixture** `schema/fixtures/golden-package/` is the single cross-language / cross-repo contract anchor: Python `jsonschema`-validates the schemas, the fixture, AND the pipeline's emitted output; Swift decodes the fixture into its Codable models; the skills repo mirrors it for the `--package` adapter.
- **Rationale**: the package is the moat — the stable contract the Claude skills consume — so freezing it correctly while there are *zero* real consumers is cheap and unblocks slice 6 (Drive), Phase 2 (context injection), and the dashboard. B1 normalizes to the shape the skills + Timeless already expect. Single-writer (Python) keeps assembly race-free and localizes Phase 2's later edits.
- **Alternatives considered**: freeze the existing `segments`/int-ms/inline-speaker shape and translate in each skill's adapter (rejected — bakes a less-normalized shape into a permanent contract); Swift seeds `metadata.json` + Python merges (rejected — two-writer merge races; the pipeline already owns the folder); meeting id = date+company+sequence (rejected — company is unknown pre-match and would re-key the folder/id on a late match); raw `SQLite3` instead of GRDB (rejected — ADR-006 chose GRDB; deviating would be an undocumented architecture change).
- **Consequences**: `INMeetingsCore` now depends on GRDB (transitive in the app; first build fetches it). Tests: Python **17** (transcript/metadata/contract), Swift **21** (ContextPackage decode, MeetingStore, JobBridge contract); `make build-mac` green; ruff clean. **⏳ PENDING LIVE VERIFICATION**: the end-to-end record → package → **index** path is untested on a real meeting (none recorded this session) — the unit/contract layer is green; a live run is the next check (HANDOFF). The skills `--package` adapter in `~/repos/claude-skills` remains a separate coordinated change (ADR-005 part 2, Phase 3).
- **Slice 5 LIVE-VERIFIED (2026-06-14)**: recorded a real Google Meet call → the DB row showed `type=call`, real ISO timestamps + 26.6s duration, `speakerCount=2`/`diarized=1`, `model_revision=ivrit-large-v3-turbo`, `capture_source_app="Google Chrome (Meet/web call)"`, `syncState=local`. ### 2026-06-14 — MVP slice 6: Drive sync — Swift-owned (amends ADR-009) + per-user dynamic backup location (refines ADR-006)
- **Agent**: Claude Code (decisions by Yuval)
- **Decision**: Per-user Google Drive write-through sync, **Swift-owned** — this **amends ADR-009** (which slotted Drive sync under the Python pipeline). The whole engine lives in `Sources/INMeetingsCore/Drive/`: PKCE OAuth (`PKCE`, `GoogleOAuth`), Keychain token storage (`DriveCredential` + `KeychainTokenStore`, with refresh-token carry-forward), live token POST (`GoogleTokenService`), refresh (`DriveTokenManager`), a Shared-Drive-aware REST client (`DriveClient`), company-first upload orchestration (`DriveSync` → `<Company>/<meeting>/`, idempotent), per-user location persistence (`DriveLocationStore`), and `DriveBackup` wired into `JobBridge` to auto-upload on pipeline `done`. The only app-target piece is the interactive sign-in + location picker (`DriveAuth` via `ASWebAuthenticationSession`, surfaced in the menu).
  - **Backup location is per-user + dynamic** (Yuval's correction, refines ADR-006): each teammate connects their own IN Venture Google account and **picks their backup Drive** in-app (persisted via `DriveLocationStore`); nothing about the destination is hardcoded. The app creates an `IN Meetings/<Company>/<meeting>/` tree under whatever Shared Drive the user selects. The only static config is the app's **OAuth client ID** (`DriveConfig` — non-secret; PKCE has no client secret).
  - **Scope** (revised 2026-06-14, per Yuval — the recordings are the point of the backup): the text package (transcript.json/.txt, metadata.json, context.md) uploads in one request each, and the **recordings (mic.wav/system.wav, + video.mov when present) stream via a resumable Drive session** (`DriveClient.uploadFileResumable`) so large files never load into memory. Drive ids + sync state fill the index's existing `driveFolderId`/`syncState` columns. A retention/size cap (ADR-010) is the remaining follow-up.
  - **OAuth client type = iOS** (PKCE + reversed-client-id custom scheme, no secret) — the right fit for `ASWebAuthenticationSession`; scope = full `drive` (to navigate the team's shared folders across machines; `drive.file` is the tighter option if ownership changes).
- **Rationale**: OAuth + token (Keychain) + the upload trigger are all Swift-native; keeping the secret in-process (never handed to a spawned subprocess) and updating the index directly beats crossing the process boundary. Putting everything except the sign-in sheet in Core keeps it unit-testable (**19 Drive tests**) and lets `JobBridge` auto-sync with no app-layer involvement.
- **Alternatives considered**: Drive under the Python pipeline (original ADR-009 — rejected: hands the OAuth token to a subprocess + crosses the SQLite-ownership boundary); a single hardcoded shared Drive (original ADR-006 framing — rejected per Yuval: per-user dynamic choice); GoogleSignIn SDK (rejected — heavy; hand-rolled PKCE is ~1 file, keeps GRDB the only dependency); a single mixed-down upload (rejected — keep mic/system separate to preserve the Me/Them split).
- **Consequences**: 19 new Core unit tests (PKCE/OAuth, token store + refresh, client request-building, sync orchestration, location store, backup no-op) — **42 Swift tests total**; `make build-mac` green. Adding a new *app-target* file requires `make gen` (XcodeGen only auto-regenerates when `project.yml` changes). **⏳ PENDING LIVE VERIFICATION**: the interactive Google sign-in is untested (needs a real account); in particular the `ASWebAuthenticationSession` presentation anchor in a menu-bar (`LSUIElement`) app is a known-tricky spot that may need a live tweak (fallback: a browser redirect via the registered URL scheme). Amends ADR-009 + ADR-006 — update those ADR files when convenient.

### 2026-06-14 — PLANNED (pre-rollout): merge tracks into a single playback file (audio, or audio+video)
- **Agent**: Claude Code (requirement from Yuval) — **NOT yet implemented; required before team rollout.**
- **Requirement**: We record dual tracks (`mic.wav` = "Me", `system.wav` = "Them") — and, for video calls, a window video — purely to make transcription/attribution clean (ADR-002/011). But anyone who wants to **listen to or watch** a meeting should get the full, natural experience: a **single audio file** (mic+system mixed) for audio meetings, or a **single video file with the audio muxed in** for video calls. They should not see or care about the separate channels.
- **Recommended approach (the "where" + "how" Yuval asked for)**:
  - **Where**: a dedicated **"playback render" step in the Swift app using AVFoundation**, kicked **right after capture (on Stop)**. It needs only the raw tracks — *not* the transcript — so it can run **concurrently with the Python transcription pipeline** (the slow part). It produces a playback artifact *alongside* the transcription tracks; both finish around the same time.
  - **Why AVFoundation (Swift), not ffmpeg (Python)**: AVFoundation muxes video+audio **and** mixes/level-balances audio natively, with **no new PATH dependency** — the pipeline deliberately avoided ffmpeg (diarize.py normalizes via soundfile/scipy). The app already owns media *capture* (AVAudioEngine, SCK), so media *production* belongs there too. Tradeoff: unlike the Python pipeline, changing merge behavior needs an app rebuild — acceptable since the mix/mux logic is stable.
  - **Output**: `audio.wav` (mic+system, **level-balanced** so neither side dominates — the two tracks are captured at different gains; match RMS/peak via `AVAudioMix`) for audio meetings; `meeting.mp4` (call-window video + the mixed audio, A/V-aligned by capture start time, via `AVMutableComposition`/`AVAssetExportSession`) for video calls.
  - **Keep the separate tracks** as the transcription inputs (the moat — dual-track Me/Them attribution depends on them); the merged file is purely the human playback artifact.
  - **Drive backup (supersedes the current interim behavior)**: once this lands, **slice 6 should upload the merged playback file** (`audio.wav` / `meeting.mp4`) as the listen/watch artifact, **instead of** the separate `mic.wav`/`system.wav` (which are transcription inputs, not the experience). The dashboard player (H4) plays the merged file.
- **Sequencing summary**: Stop → (a) spawn the transcription pipeline [existing] and (b) kick the AVFoundation render [new], concurrently → both land in the meeting folder → the merged file is what's uploaded (slice 6) and played (H4).
- **Open**: stereo placement (Me left / Them right) vs a balanced mono mix — default to balanced; revisit if users want channel separation. Retention (ADR-010): once the merged file exists + is uploaded, the raw tracks can be pruned locally.

### 2026-06-14 — Phase 2 slice 1: calendar context assembler — Swift fetch / Python transform (amends ADR-004)
- **Agent**: Claude Code (design by Yuval; Yuval delegates technical/architecture decisions, sets behavior)
- **Decision**: First Phase-2 sub-slice = **Google Calendar only**. Swift fetches candidate events (slice-6 Google OAuth + new `calendar.events.readonly` scope) and writes `<meeting>/context.input.json` *before* spawning the pipeline; the Python pipeline matches the event (max time-overlap with the recording window), splits internal/external by the **signed-in account's domain**, builds the post-correction vocab (`context.vocab.json`, consumed by the existing `load_vocab`→`postcorrect` hook), renders `context.md` (priors, with the priors/content wall), and merges fields into `metadata.json` (`meeting.title`/`calendar_event_id`, `attendees[]` with side, `company{matched:false}`, `context.sources.calendar`). Personal names are priors only (not rewritten, to avoid corrupting the transcript). Post-correction was hardened to **whole-token** (boundary-aware) replacement.
