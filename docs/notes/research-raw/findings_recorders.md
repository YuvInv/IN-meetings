# No-bot native macOS meeting recorders — competitive teardown

## Landscape map

The category splits into three architectures: (a) **cloud-ASR no-bot recorders** — Granola, Notion AI Meeting Notes, ChatGPT Record Mode, Circleback desktop, Notta desktop, Jamie, Fellow botless; (b) **local-ASR no-bot recorders** — MacWhisper, Superwhisper, Hyprnote (rebranded **Char** in 2025), Meetily, plus a long tail (Thoth, VocAI, Meetly, Macnotetaker, Whisper Notes, SpeakApp) ([heymumble roundup](https://heymumble.com/blog/local-ai-meeting-note-takers-mac), [meetily.ai/bot-free](https://meetily.ai/bot-free)); (c) **driver/overlay outliers** — Krisp (virtual audio devices) and Cluely (GPU-hook overlay). Named tools that turned out NOT to be in-category: **Spoke** ([spoke.app](https://spoke.app/)) is now a meeting-bot API platform, not a Mac recorder; **Aiko** ([sindresorhus.com/aiko](https://sindresorhus.com/aiko)) is file/mic-only local Whisper transcription — no system-audio capture, no live transcription, no speaker detection; **Fathom**'s "bot-free" Zoom mode actually uses Zoom's Live Stream API, requires linking your Zoom account, and posts a recording notice into meeting chat — not invisible ([Fathom help](https://help.fathom.video/en/articles/295552)).

## Per-tool findings

**Granola** (the category leader). Detection: calendar event (Google/Outlook) opens a notepad at meeting time, plus mic-in-use detection for unscheduled calls; a call started within 15 min after a calendar event is auto-matched to it; notification copy adapts per app ("Huddle detected" for Slack, "Call detected" for FaceTime/WhatsApp) ([Granola notifications doc](https://docs.granola.ai/help-center/taking-notes/notifications)). UI: menu-bar/notepad hybrid, prompt-to-record (not silent auto-record). Capture: mic + system audio via the macOS "Screen & System Audio Recording" privacy pane; it captures the **combined system mix and "cannot isolate individual applications"** ([transcription doc](https://docs.granola.ai/help-center/taking-notes/transcription)). Tracks: two channels rendered as grey bubbles (system = "Them") and green bubbles (mic = "Me"); desktop cannot name individual remote speakers — "the models don't yet support live diarization." Transcription: **cloud** — Deepgram + AssemblyAI, audio deleted after real-time transcription; enhancement via GPT-4o/Claude ([Granola security](https://www.granola.ai/security), [review](https://zackproser.com/blog/granola-ai-review)). Languages: ~10 on desktop — **no Hebrew**; enabling multi-language mode **disables the custom-vocabulary feature** ([multi-language doc](https://docs.granola.ai/help-center/customising-granola/multi-language)). Pain: macOS 13+ (best ≥14.2); troubleshooting doc blames default-device mismatch, Bluetooth/USB dropouts ("if transcription stops after a few minutes, try built-in audio"), and requires both mic + Screen & System Audio permissions ([troubleshooting](https://docs.granola.ai/help-center/troubleshooting/transcription-issues)); reviewers flag garbled numbers and no per-speaker names ([tl;dv review](https://tldv.io/blog/granola-review/)).

**MacWhisper** (closest to our architecture). Detection: monitors a configurable app list (Zoom, Teams, Slack, Webex, Discord, Chrome/Safari/Arc/Brave/Edge/Firefox, WhatsApp, Telegram…) and fires a notification prompting record; **FaceTime is excluded by macOS privacy restrictions**. Permissions: Screen Recording + Microphone. Auto meeting recording is **beta with explicit data-loss warnings** ("for critical meetings we recommend manual"); feature absent from the App Store build ([MacWhisper support doc](https://macwhisper.helpscoutdocs.com/article/30-record-meetings)). Transcription: local Whisper, speaker labels assignable post-hoc.

**Superwhisper**: dictation-first; Meeting Mode records app audio from Zoom/Teams/Meet, transcribes **locally**, speaker separation off by default and shown only in segment view ([superwhisper.com/docs/modes/meeting](https://superwhisper.com/docs/modes/meeting), [meeting-transcription page](https://superwhisper.com/meeting-transcription)). Manual start; no auto-detection comparable to Granola.

**Hyprnote/Char & Meetily** (open-source references worth mining). Hyprnote: Tauri app, mic + system audio, on-device fine-tuned Whisper + local LLM, BYOK optional ([SourceForge mirror](https://sourceforge.net/projects/hyprnote.mirror/), [anarlog/fastrepl repo](https://github.com/fastrepl/anarlog)). Meetily: MIT-licensed Tauri/Rust, whisper.cpp + Parakeet, local summaries via Ollama ([github.com/Zackriya-Solutions/meetily](https://github.com/Zackriya-Solutions/meetily)).

**Krisp**: no bot, but architecture is exactly what we must avoid — installs **Krisp Microphone/Speaker virtual devices** that you select inside the meeting app ([Krisp help](https://help.krisp.ai/hc/en-us/articles/4402174576402-How-Krisp-Microphone-and-Krisp-Speaker-work)). On-device transcription is **English-only**; 15 other languages are server-side; **no Hebrew** ([Krisp note taker](https://krisp.ai/ai-note-taker/)).

**Cluely**: real-time overlay rendered "via low-level GPU hooks (Metal on macOS)" to stay invisible in screen shares; system audio → STT → cloud LLM ([docs.cluely.com/feature/undectability](https://docs.cluely.com/feature/undectability), [Shadecoder teardown](https://www.shadecoder.com/blogs/zoom-cannot-detect-cluely-here-s-why-technical-breakdown-risks)). Cloud-dependent and reputationally toxic; relevant only as proof the overlay UX works.

**Platform entrants (squeeze risk)**: Notion AI Meeting Notes records system audio + mic from the Notion desktop app, `/meet` to start, no bot, cloud transcription ([Notion help](https://www.notion.com/help/ai-meeting-notes)). ChatGPT Record Mode (macOS app, Plus/Pro/Business) captures mic + system audio via the same Screen & System Audio Recording pane, 120-min cap, server-side processing ([OpenAI help](https://help.openai.com/en/articles/11487532-chatgpt-record), [MacRumors 2025-07-17](https://www.macrumors.com/2025/07/17/chatgpt-plus-gets-record-mode-macos/)).

## Capture mechanism — the decisive technical finding

Three routes exist ([Recall.ai engineering blog](https://www.recall.ai/blog/how-to-access-to-system-audio)): virtual drivers (excluded by constraint; Krisp-style), **ScreenCaptureKit** (audio tied to a capture session; needs Screen Recording permission; captures the whole mix incl. notifications), and **Core Audio process taps** (`CATapDescription`, API macOS 14.2, practical from 14.4 — see [AudioCap sample](https://github.com/insidegui/AudioCap) and [NSAudioCaptureUsageDescription](https://developer.apple.com/documentation/bundleresources/information-property-list/nsaudiocaptureusagedescription)). Taps need only **"System Audio Recording Only"** TCC — no Screen Recording permission, no admin password, no restart, and crucially **no Sequoia/Tahoe monthly re-approval nag**, which applies to screen-capture apps ([9to5Mac](https://9to5mac.com/2024/08/14/macos-sequoia-screen-recording-prompt-monthly/), [Apple user guide](https://support.apple.com/guide/mac-help/control-access-screen-system-audio-recording-mchld6aa7d23/mac)). Audio Hijack 4.x is the production proof: "no need to adjust security settings, install anything, nor enter your administrator password" ([Rogue Amoeba](https://rogueamoeba.com/support/knowledgebase/?showArticle=Misc-ARK-Audio-Capture-Details&product=Audio+Hijack)). Taps also allow **per-process capture** (tap only zoom.us / Chrome / Teams / Slack), eliminating the Slack-ping-in-transcript problem Granola explicitly can't solve. A local purple mic indicator still appears — visible only on the recorder's own Mac, not to call participants. An Electron-context comparison confirms taps avoid the purple "screen recording" Control-Center indicator that the Chromium/SCK path triggers ([stronglytyped.uk](https://stronglytyped.uk/articles/recording-system-audio-electron-macos-approaches)).

## Detection patterns

Best-in-class = Granola's **calendar + mic-in-use + app-aware notification** stack. Implementation reality: `kAudioDevicePropertyDeviceIsRunningSomewhere` detects mic activity but **Bluetooth microphones (AirPods) do not report status correctly** ([Apple dev forums](https://forums.developer.apple.com/forums/thread/741026)), and there is no public API to know *which* app holds the mic ([tutorialpedia](https://www.tutorialpedia.org/blog/is-there-an-api-to-check-if-mac-s-microphone-or-video-camera-is-in-use/)) — so combine: running-process/app list (MacWhisper-style), CoreAudio mic heuristic, and calendar windows with Granola's 15-minute matching rule.

## Failure modes to design against

1. **Silent missed recordings** after the monthly screen-recording re-approval lapses (any SCK-based tool); Jordi Bruin shipped a whole utility, Amnesia, just to suppress the nag ([tinyapps](https://tinyapps.org/blog/202409180700_disable_sequoia_nag.html)). Avoided entirely by taps. 2. **Bluetooth/USB device dropouts** mid-meeting and default-device mismatch (Granola's #1 troubleshooting items). 3. **Echo/double transcription** without headphones — mic re-captures speaker output; needs an AEC stage fed by both streams (Recall.ai). 4. **Auto-record unreliability** — even MacWhisper labels it beta with data-loss warnings; provide manual start + ring-buffer so the first minutes aren't lost. 5. **FaceTime is uncapturable** — set expectations. 6. **Numbers/entity errors** (Granola complaint) — vocabulary biasing directly addresses this. 7. Notification prompts invisible over fullscreen meeting windows — prefer a floating panel/menu-bar state change over Notification Center.

## Verdict

Best-in-class composite: Granola's detection + prompt UX, Core Audio process taps per-process capture (Audio Hijack-proven, nag-free), MacWhisper/Superwhisper's local Whisper privacy posture, two-channel Me/Them transcripts. Nobody combines local processing + Hebrew + vocabulary biasing — Granola has no Hebrew and disables custom vocabulary in multi-language mode; Krisp's on-device ASR is English-only. The strategic bet is unoccupied territory.
