# Topic D — macOS 26 capture APIs: findings

## 1. ScreenCaptureKit (SCK) on macOS 26

- SCK captures three media types per stream: screen, system audio (`SCStreamConfiguration.capturesAudio`, macOS 13+, recommended 48 kHz stereo) and, since macOS 15, microphone (`captureMicrophone` + `microphoneCaptureDeviceID`); app-audio and mic arrive as separate sample buffers distinguished by output type on the same stream ([Apple: capturesAudio](https://developer.apple.com/documentation/screencapturekit/scstreamconfiguration/capturesaudio?language=objc), [captureMicrophone](https://developer.apple.com/documentation/screencapturekit/scstreamconfiguration/capturemicrophone), [Capturing screen content in macOS](https://developer.apple.com/documentation/ScreenCaptureKit/capturing-screen-content-in-macos)).
- **There is no true audio-only SCK stream**: an `SCContentFilter` must name a screen target (display/window/apps) even if you ignore video frames, and the whole stream is gated on the Screen Recording TCC ([Recall.ai: How to get access to system audio on macOS](https://www.recall.ai/blog/how-to-access-to-system-audio)). `excludesCurrentProcessAudio` removes your own app's audio ([Apple doc](https://developer.apple.com/documentation/screencapturekit/scstreamconfiguration/excludescurrentprocessaudio)); app-based filters can include/exclude specific apps' audio ([Creavit SCK audio guide](https://creavit.studio/blog/screencapturekit-audio-recording-mac-guide)).
- `SCRecordingOutput` (macOS 15+) records direct-to-file: configure `SCRecordingOutputConfiguration` (outputURL, file type, codec), attach to the stream, get start/error/finish callbacks via `SCRecordingOutputDelegate` ([Apple: SCRecordingOutput](https://developer.apple.com/documentation/screencapturekit/screcordingoutput), [WWDC24 HDR/SCK session](https://developer.apple.com/videos/play/wwdc2024/10088/)). It is video-recording oriented; for audio-track files we'd still write our own AVAudioFile path.
- macOS 26 (Tahoe) added SCK "advanced screenshots" (screenshot output APIs) but **no relevant deprecations or new audio-recording APIs surfaced** in release notes/search; user-facing Tahoe additions (Record Selected Window, HDR capture toggle) ride on the same APIs ([macOS 26 release notes](https://developer.apple.com/documentation/macos-release-notes/macos-26-release-notes), [crates.io screencapturekit notes on 26.0 screenshots](https://crates.io/crates/screencapturekit)). Note: Apple Community reports of generally muffled/quiet audio after Tahoe 26.0/26.3 updates exist — regression-test audio on each 26.x ([thread](https://discussions.apple.com/thread/256140785)).

## 2. Core Audio process taps — the better fit

- `CATapDescription` + `AudioHardwareCreateProcessTap` (API macOS 14.2+, sample code targets 14.4+) tap audio of specific processes (PID → AudioObjectID via `kAudioHardwarePropertyTranslatePIDToProcessObject`) or the whole system (empty process list + `isExclusive=true`, inverted logic). The tap is added to an **aggregate device** (`kAudioAggregateDeviceTapListKey`), read via `AudioDeviceCreateIOProcIDWithBlock`, written to `AVAudioFile` ([Apple: Capturing system audio with Core Audio taps](https://developer.apple.com/documentation/CoreAudio/capturing-system-audio-with-core-audio-taps), [insidegui/AudioCap](https://github.com/insidegui/AudioCap), [AudioTee write-up](https://stronglytyped.uk/articles/audiotee-capture-system-audio-output-macos)).
- **Permission**: `NSAudioCaptureUsageDescription` in Info.plist; the user grant lands in Privacy & Security under "Screen & System Audio Recording", which explicitly supports allowing apps to record "both your screen and audio, **or just your audio**" — the audio-only ("System Audio Recording Only") category; this guide covers through macOS Tahoe 26 ([Apple Support](https://support.apple.com/guide/mac-help/control-access-screen-system-audio-recording-mchld6aa7d23/mac)). **Screen Recording permission is NOT required** ([AudioCap README](https://github.com/insidegui/AudioCap)).
- **Gotcha**: there is **no public API to check or request** this permission; AudioCap uses private TCC SPI (build-flag removable; the OS prompts on first tap use). Fine for internal distribution; an App Store risk ([AudioCap](https://github.com/insidegui/AudioCap)).
- The system can deliver a mono or stereo mixdown of the tap; custom channel layouts aren't guaranteed (a 5.1 source case needed matching aggregate channels) ([maven.de CoreAudio Taps for Dummies, Apr 2025](https://www.maven.de/2025/04/coreaudio-taps-for-dummies/)). Tapping shows a **purple recording indicator** in the menu bar — local-only, invisible to remote participants ([maven.de](https://www.maven.de/2025/04/coreaudio-taps-for-dummies/)).
- **SCK vs taps for our use**: taps give remote-participant audio as its own clean track, per-process targeting (Zoom/Teams/browser only — notification dings and music excluded), and avoid Screen Recording TCC entirely. SCK's audio is simpler but drags in the screen-permission UX. Recall.ai (who build exactly this product category) confirm SCK "ties audio to screen capture" and that taps are the no-screen-permission path, while warning tap documentation is sparse ([Recall.ai](https://www.recall.ai/blog/how-to-access-to-system-audio)).

## 3. Mic capture and the echo problem

- Capture the mic with `AVAudioEngine.inputNode`. Voice-processing IO (`setVoiceProcessingEnabled(true)`) brings Apple's AEC but with documented macOS side effects: **expected gain reduction** on mic and output (Apple engineer confirmation), **ducking of other audio** (i.e., it can duck the very Zoom output we're recording via the tap), auto-enabling VP on the paired output node, and aggregate-construction failures with mismatched devices (AirPods mic + built-in speakers) ([Apple forum: AUVoiceProcessing echo cancellation](https://developer.apple.com/forums/thread/733733), [volume issue thread](https://developer.apple.com/forums/thread/721535), [forum 128518](https://developer.apple.com/forums/thread/128518)). macOS 14+ exposes `voiceProcessingOtherAudioDuckingConfiguration` (advanced ducking on/off, duckingLevel min/mid/max) to tame this ([Apple doc](https://developer.apple.com/documentation/avfaudio/avaudioinputnode/voiceprocessingotheraudioduckingconfiguration), [WWDC23 What's new in voice processing](https://wwdcnotes.com/documentation/wwdcnotes/wwdc23-10235-whats-new-in-voice-processing/)).
- **Recommended mitigation**: record the mic raw (no VP) by default; since the tap track is a perfect far-end reference, run software AEC offline/in-post (or only enable VP when built-in speakers are the active output). Headphone use makes echo moot. Whether macOS VPIO's reference includes other processes' output (hardware loopback on Apple Silicon) is empirically disputed — validate on the M3 before relying on it ([OpenAI community notes on Apple audio](https://community.openai.com/t/audio-notes-for-openai-realtime-on-apple-platforms/1108404)).

## 4. TCC permission set and the re-approval nag

- Needed: **Microphone**, **System Audio Recording Only** (audio taps), **Calendar** (EventKit), **Automation/Apple Events** per browser for tab URLs (`NSAppleEventsUsageDescription`). **Not needed**: Screen Recording, Accessibility (if using AppleScript instead of AX).
- The macOS 15+ re-approval prompt ("…requesting to bypass the system private window picker and directly access your screen and audio… Allow For One Month") went weekly → monthly in 15.0, frequency further relaxed in 15.1, persists on macOS 26, and is undisableable outside MDM ([MacRumors](https://www.macrumors.com/2024/08/15/macos-sequoia-screen-recording-app-permissions/), [MacRumors 15.1](https://www.macrumors.com/2024/10/07/apple-screen-recording-popup-update/), [tinyapps](https://tinyapps.org/blog/202409180700_disable_sequoia_nag.html), [Screenify 2026 guide](https://www.screenify.studio/blog/2026-04-23-macos-screen-recording-permissions)). It is tracked per-app in `~/Library/Group Containers/group.com.apple.replayd/ScreenCaptureApprovals.plist` — i.e., keyed to **screen capture**; no reports tie it to audio-tap-only ("System Audio Recording Only") apps, which is a major UX argument for taps. Apps using `SCContentSharingPicker` also avoid it, but that's irrelevant if we skip SCK ([TidBITS](https://tidbits.com/2024/09/23/how-to-avoid-sequoias-repetitive-screen-recording-permissions-prompts/)).

## 5. Detection signals

- `kAudioDevicePropertyDeviceIsRunningSomewhere` (mic in use by anyone): works for built-in and wired mics, but **always returns false for Bluetooth mics** (AirPods) — Apple engineer acknowledged as a bug, unresolved ([Apple forum 741026](https://forums.developer.apple.com/forums/thread/741026)). No public API identifies *which* process uses the mic ([tutorialpedia overview](https://www.tutorialpedia.org/blog/is-there-an-api-to-check-if-mac-s-microphone-or-video-camera-is-in-use/)). Use it as one signal among several (frontmost app via `NSWorkspace`, running meeting apps, calendar window, browser tab URL), never alone.
- Browser tab URL: AppleScript (`URL of active tab of front window`) works for Safari and Chrome and needs the Automation TCC, not Accessibility ([gist of per-browser scripts](https://gist.github.com/vitorgalvao/5392178), [Itsuki Medium 2-ways article](https://medium.com/@itsuki.enjoy/swift-macos-get-browser-opened-tab-url-2-ways-e6722fb5998d)). The AX route (`kAXURLAttribute` on `AXWebArea`) is unreliable on Chromium: the accessibility tree is lazily built and requires setting `AXEnhancedUserInterface` (breaks window positioning) or `AXManualAccessibility` (broken in some Electron versions) ([Chromium a11y docs](https://www.chromium.org/developers/design-documents/accessibility/), [Electron #37465](https://github.com/electron/electron/issues/37465)). Use AppleScript; Firefox has no script support (accept degraded detection there).

## 6. Floating NSPanel over fullscreen meetings

`NSPanel` with `.nonactivatingPanel` styleMask, `isFloatingPanel`, level `.floating`+, and `collectionBehavior = [.canJoinAllSpaces, .fullScreenAuxiliary]` is the working recipe for floating over fullscreen Zoom/Meet on current macOS; `.fullScreenAuxiliary` is what permits joining fullscreen Spaces, `.canJoinAllSpaces` makes it follow Space switches ([Apple: nonactivatingPanel](https://developer.apple.com/documentation/appkit/nswindow/stylemask-swift.struct/nonactivatingpanel), [fazm.ai NSPanel patterns](https://fazm.ai/blog/swiftui-floating-panel)).

## Recommended architecture

Dual-track, SCK-free: (1) **system/remote track** via Core Audio process tap (system-wide-exclusive, or per-process pinned to the detected meeting app) → aggregate device → IOProc → AVAudioFile; (2) **mic track** via AVAudioEngine, raw by default, offline AEC using the tap track as reference (VP IO with ducking set to `.min` as fallback for speaker users). Permissions: Mic + System Audio Recording Only + Automation + Calendar. No Screen Recording TCC, no monthly nag, and separate tracks give free two-party diarization for Hebrew/English ASR.
